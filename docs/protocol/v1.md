# vibe-terminal Protocol v1

All WebSocket messages use this JSON envelope:

```json
{
  "type": "start_session",
  "session_id": "sess-1",
  "payload": {}
}
```

The server accepts two WebSocket entry points:

- `/ws/agent`: authenticated by device credentials.
- `/ws/web`: authenticated by the administrator session cookie.

Required message types:

- `agent_hello`
- `heartbeat`
- `sync_sessions`
- `start_session`
- `session_started`
- `stdin`
- `resize`
- `stdout`
- `session_exit`
- `close_session`
- `subscribe_session`
- `session_state`
- `error`

## File transfer extension

An agent that supports file operations declares it in `agent_hello`:

```json
{
  "type": "agent_hello",
  "payload": {
    "capabilities": ["fs"]
  }
}
```

The server only forwards fs messages to agents that declared the `fs` capability.

### Request correlation

Every fs request envelope carries a `request_id` set by the server. The agent
must echo the same `request_id` on the reply envelope. The server matches the
reply to the pending call by `request_id`; replies without a known
`request_id` are dropped.

```json
{
  "type": "fs_list",
  "request_id": "req-1",
  "payload": { "path": "/tmp" }
}
```

### Messages

| Type | Direction | Payload fields |
| --- | --- | --- |
| `fs_list` | server → agent | `path` |
| `fs_list_result` | agent → server | `path`, `entries` (`name`, `is_dir`, `size`, `mode`, `modified_at`) |
| `fs_read` | server → agent | `path`, `offset`, `length` |
| `fs_read_result` | agent → server | `data` (base64), `eof`, `file_size` |
| `fs_write_open` | server → agent | `upload_id`, `path`, `size`, `overwrite` |
| `fs_write_opened` | agent → server | `upload_id` |
| `fs_write_chunk` | server → agent | `upload_id`, `offset`, `data` (base64) |
| `fs_write_ack` | agent → server | `upload_id`, `offset` |
| `fs_write_close` | server → agent | `upload_id`, `total_size` |
| `fs_write_result` | agent → server | `upload_id` |
| `fs_error` | agent → server | `code`, `message` |

`fs_error` replaces the normal result message when an operation fails and
carries the same `request_id` as the failed request.

### Limits

- Chunk size for `fs_read` / `fs_write_chunk`: 256 KiB.
- Server waits at most 30s for each agent reply, then fails with `timeout`.
- At most 4 concurrent fs operations per device on the server side.
- The agent drops uploads with no activity for 60s and deletes their
  temporary files.

### Error codes

`fs_error.code` is one of:

- `not_found`
- `permission_denied`
- `not_a_file`
- `not_a_directory`
- `already_exists`
- `invalid_path`
- `invalid_request`
- `io_error`
